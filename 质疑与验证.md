## 已实现功能

### A1. 三层意图路由

**声称**: L1 关键词 (0ms) → L2 Embedding 原型 (~15ms) → L3 兜底

**质疑**: L3 目前还是关键词 (`resolveNewsIntent`)，和 L1 用同一套 `IntentKeywords`。等于只有两层——L1 和 L2，L3 是假三层。

- [x] L1 关键词精确命中 → ✅ 已验证 (IntentRouterTest, 18 tests)
- [x] L2 Embedding 原型相似度 → ✅ 已验证 (test_intent.py, 21 tests)
- [ ] L3 是真正的 LLM 分类，不是关键词 → ❓未验证

**结论**: 名义三层，实际两层。L3 是标记为"TODO"的关键词回退。

### A2. L2 Embedding 原型质量

**声称**: 120 条原型短语覆盖 9 类意图，能理解"有啥新鲜事"等口语表达。

**质疑**:
1. 120 条是我手写的，不是从真实用户数据中提取的。"最近闹啥呢"这类表达没覆盖到怎么办？
2. `cosine_similarity("最近闹啥呢", "最近有什么新闻")` 的分数是多少？能超过 0.65 阈值吗？
3. "嗯"被精确匹配到 casual_chat 置信度 1.0——但"嗯"也可能是"确认看第 N 条"。单字歧义无法用 Embedding 解决。

- [ ] 用真实用户数据验证原型覆盖率 → ❓未验证
- [ ] 验证边界 case 的置信度分布 → ❓未验证
- [ ] 积累 `L2_L3_DISAGREE` 日志并统计准确率 → ❓未验证

**结论**: 原型质量未经真实数据验证。21 个测试全是手工构造的理想输入。

### A3. Tool Calling 协议

**声称**: LLM 可自主调用 3 个工具，ReAct 循环，3 轮上限，括号深度匹配解析。

**质疑**:
1. 全用 mock 测试——`toolLoop` 从未在真实 LLM 上跑过。LLM 会严格输出 `[TOOL:name] {...}` 格式吗？如果 LLM 在花括号里写中文冒号 `：` 而不是英文 `:`，`parseToolArgs` 会解析失败吗？
2. 3 轮上限是拍脑袋的。用户说"找三条新闻分析再查后续"需要 5+ 轮。
3. `findMatchingBrace` 在 LLM 真实输出"工具结果: {data: {nested: {...}}}"时表现如何？没测过。

- [x] findMatchingBrace 边界测试 → ✅ 已验证 (ToolLoopTest, 19 tests)
- [ ] 用 deepseek 模型真实跑一次 → ❓未验证
- [ ] 统计多轮任务的实际轮次需求 → ❓未验证

**结论**: 解析层经过边界验证（嵌套、转义、中英文混合、字符串内花括号）。但仍未用真实 LLM 测试过格式兼容性。

### A4. Tool 实现质量

**声称**: `FetchNewsTool`、`AnalyzeArticleTool`、`TrackFollowUpTool` 三个工具。

**质疑**:
1. `TrackFollowUpTool.execute()` 每次调用都 `newsService.fetchAll()`——同一轮 toolLoop 里调两次（先 fetch_news 再 track_follow_up），RSS 被请求了两次。浪费。
2. `AnalyzeArticleTool.execute()` 只收 title + content，但不知道当前快照里有没有这条。tool 说"分析"，但实际可能分析了一条不存在于当前系统的新闻。
3. `FetchNewsTool` 不支持分页、不支持日期范围、不支持按源筛选——只能拉全部再过滤。

- [x] Tool 接口定义清晰 → ✅
- [ ] TrackFollowUp 不应重复 fetch → ⚠️存疑
- [ ] Tool 需要知道当前快照状态 → ⚠️存疑

**结论**: Tool 能用，但不够精致。某些场景会重复请求或操作不存在的条目。

### A5. 置信度反问

**声称**: `confidence < 0.55` 时反问用户而非硬猜。

**质疑**:
1. 0.55 是哪来的？是调参出来的吗？有 A/B 数据吗？——没有，拍脑袋的。
2. 低置信度 + DEFAULT = 不反问，直接走 buildAiReply。但如果 L2 返回 "news_overview" (0.52)，这不属于 DEFAULT，会触发反问——但用户明明说的是"看新闻"变体。反问反而多此一举。
3. 澄清话术有三条，但"你是不是想看新闻，还是随便聊聊"这句话在新闻 Bot 的场景里很奇怪——就没见人说过"随便聊聊"。

- [ ] 0.55 阈值需要验证 → ❓未验证
- [x] DEFAULT 不过问的逻辑正确 → ✅
- [ ] 澄清话术是否自然 → ⚠️存疑

**结论**: 逻辑框架对，但没有数据支撑阈值和话术质量。

### A7. 推送去重

**声称**: MD5 标题 hash 签名，重复就跳过。

**质疑**:
1. 如果 12 条新闻里 11 条一样、1 条更新——签名不同→照推。但用户看到 11 条重复 + 1 条新的，体验很差。MD5 去重是"全不变才不推"，不是"只推新的"。
2. `computeNewsSignature` 在 `DailyBotSchedulerTest` 里没被直测。覆盖率是虚的。
3. `lastPushSignature` 是 `volatile` 内存变量——bot-server 重启后丢失，重启后第一条推送一定不会被 skip。这可能是故意的（重启后应该推一次），但没文档说明。

- [x] MD5 签名正确 → ✅
- [ ] 部分更新时只推增量 → ❓未验证（目前是全量推）
- [ ] 重启行为有文档说明 → ❓未验证

**结论**: 功能正确，但粒度太粗——只区分"完全相同"和"不完全相同"。

## 规划中的功能

### B1. 双轨 Memory 架构

**声称**: 新闻浏览用交互深度评分，追踪用话题簇，社交用 RIF。

**质疑**:
1. 单用户 Bot 一个月产生约 100 条浏览记忆。一个 ArrayList 排序取 top-3 就够了——交互深度公式是过度设计。
2. 社交记忆的用户画像需要大量闲聊数据才能生效。如果用户 80% 的消息是"今日热点"/"3"/"分析"，画像永远是空的。
3. ChromaDB 存 100 条向量是杀鸡用牛刀——一个 JSON 文件 + 内存计算 cosine similarity 就够。

- [ ] 数据量是否足够支撑评分系统 → ⚠️存疑
- [ ] 社交记忆是否有足够的输入数据 → ⚠️存疑
- [ ] ChromaDB 是否为过度选型 → ⚠️存疑

**结论**: 架构是对的，但复杂度是为百倍于实际的数据量设计的。

---

### B2. 追踪记忆三层触发

**声称**: Layer 1 被动匹配 + Layer 2 显式追踪 + Layer 3 主动查询。

**质疑**:
1. 如果用户标记了 5 个追踪话题，每天 RSS 更新 12 条，每天 60 次向量匹配。一个人用的话，这些匹配每次都在同一个进程里跑——根本没必要异步。
2. Layer 1 和 Layer 2 可能命中同一条新新闻，需要去重逻辑。如果去重漏了，用户收到两条推送讲同一条新闻——非常蠢。
3. Layer 3 调"搜索"——但搜索 tool 不存在。目前 `track_follow_up` 只查本地 ChromaDB。要接搜索需要新的 tool 和新的 API 依赖。

- [ ] 重复推送去重 → ⚠️存疑
- [ ] Layer 3 的搜索 tool 存在吗 → ❌不合理（还没做）
- [ ] 每天 60 次匹配的性能 → ❓未验证

**结论**: 逻辑对，但缺关键组件（搜索 tool），且有重复推送风险。

---

### B3. 话题簇 chain

**声称**: 追踪记忆以 chain 形式积累事件时间线，匹配到的新新闻自动加入。

**质疑**:
1. 如果两个追踪话题重叠了怎么办？"英伟达芯片"和"AI 芯片"追踪标记同时命中同一条新新闻——chain 里会重复加入吗？
2. chain 可能永远不会很长——单用户追踪 5 个话题，每月每个命中 1 次，一年 chain 长度 12。不值得用 ChromaDB。
3. 链的展示："你关注的英伟达 B300 进展: 6/3 发布 → 6/10 延期 → 6/25 量产"——这个格式很好，但一个 JSON 文件就够存了。

- [ ] 重叠话题去重 → ❓未验证
- [ ] ChromaDB 是否必要 → ⚠️存疑

**结论**: 话题簇设计优美，但实现选型可以更轻。

---

### B4. 社交记忆 RIF 评分

**声称**: RIF = 0.4×时近性 + 0.3×相关性 + 0.3×频率，三级流转。

**质疑**:
1. 照搬聊天 Bot 的记忆公式。聊天 Bot 的"频率"有意义——用户反复提同一件事说明重要。新闻 Bot 的社交交互是"早上好"、"谢谢"、"哦"——"哦"的频率高说明什么？
2. 三级流转假设记忆有"升级"的必要。但一个新闻 Bot 用户的社交数据——"打招呼"、"说谢谢"、"偶尔抱怨"——哪一条需要从情景记忆升级到语义记忆？
3. 情感差异衰减（正面慢、负面快）需要一个情感分类器来判断"这条社交记忆是正面的还是负面的"。这个分类器存在吗？

- [ ] RIF 在新闻 Bot 场景是否适用 → ⚠️存疑
- [ ] 三级流转是否有实际需要 → ⚠️存疑
- [ ] 情感分类器存在吗 → ❌不合理（没有）

**结论**: 把聊天 Bot 的社交记忆方案搬过来，三个核心假设（频率有意义、需要升级、有情感分类器）在新闻 Bot 场景下都不成立。

---

### B5. Memory 优化五项

**声称**: ID 去重、单向量多索引、链存快照、意图分流检索、共享基类。

**质疑**:
1. 五项优化是为"三条轨道、大量数据"的场景设计的。如果实际数据量只有 100 条，这些优化大部分是负优化——增加了代码复杂度，但省下的检索时间和存储空间微不足道。
2. "意图分流检索"假设意图分类准确率够高——但 L2 准确率没有真实数据支撑。
3. "共享基类"的代价是基类可能膨胀。如果三个轨道最终只写 200 行业务代码，共享基类只省了 50 行——不值得。

- [ ] 数据量是否够大以至于需要这些优化 → ⚠️存疑
- [ ] 五项优化的整体收益是否为正 → ⚠️存疑

**结论**: 优化之前应该先确认问题真的存在。目前的问题是"数据太少"，不是"数据太多检索太慢"。

---

### B6. 卡片超采样渲染

**声称**: 2x BufferedImage 渲染后缩放，提升清晰度。

**质疑**:
1. 当前 1x 渲染在微信上已经够清晰了。2x 消耗 4x 内存——1120×2000 的卡变成 2240×4000 = ~35MB 的单张 BufferedImage。如果多个请求并发渲染，可能 OOM。
2. DPR 对 HTML 截图的影响远大于 Java 2D——因为 HTML 渲染依赖浏览器 text rendering engine，低 DPI 时字体边缘模糊。Java 2D 的 `RenderingHints.ANTIALIAS_ON` 已经在 1x 下提供抗锯齿，2x 的增益有限。

- [ ] 1x 和 2x 的实际视觉差异有多大 → ❓未验证
- [ ] 并发渲染的内存安全 → ❓未验证

**结论**: 真寻日报抄过来的优化，但 Java 2D 和 HTML 渲染的本质不同——Java 2D 已经有 antialiasing，2x 的边际增益可能很小。

---

### B7. 自动注册推送订阅

**声称**: 用户首次交互自动写入订阅表，不需要手动配置目标。

**质疑**:
1. 当前服务是私聊 Bot，用户只有一个——"自动注册"就是一行 `if (first_message) subscribe(user)`。不需要表、不需要调度器改造——一个 boolean 标记就够了。
2. 如果以后有多个用户，"谁订阅了什么"——这个问题需要解决，但"以后"是什么时候？

- [ ] 单用户场景下是否需要订阅表 → ⚠️存疑

**结论**: 为多用户场景设计的订阅系统，但当前只有一个用户。

### B8. improve_prototypes.py 迭代闭环

**声称**: 每月跑一次 → `grep DISAGREE` → 生成 LLM prompt → 补原型 → L2 覆盖率上升。

**质疑**:
1. 需要用户量支撑。一个用户每天 10 条消息，一个月 300 条。如果 L2 置信度高的占 80%，DISAGREE 只有 60 条。60 条里去重后大概 20 条不同的 expression。20 条里 LLM 生成的补充原型大概 5-10 条有效。整个流程的人力投入（跑脚本 + 审校 + 部署）约 30 分钟/月——对 5-10 条原型的增益来说，投入产出比存疑。
2. 用户太少 → DISAGREE 样本太少 → LLM 分析不出模式 → 原型补不准 → 永远到不了稳态。
3. 如果长期不跑，脚本就变成了 dead code。

- [ ] 是否有足够的 DISAGREE 样本 → ⚠️存疑
- [ ] 人力投入产出比 → ⚠️存疑
- [ ] 是否需要"最少 N 条样本"的门槛 → ❓未验证

**结论**: 设计好，但依赖用户量和定期维护——两个条件在当前阶段都不满足。

## 全局性质疑

### C1. 整个项目是为多少用户设计的？

```
代码里的多用户痕迹:
- ConversationStateManager: Map<conversationId, ...>  ← 支持多用户
- DailyBotScheduler: 推送目标列表
- Memory 规划: 三层评分 + ChromaDB
- 订阅规划: 推送订阅表

实际用户数: 1
```

大部分复杂度是为"多用户"场景设计的，但只有一个用户在用一个微信私聊 Bot。如果把多用户假设去掉，Memory 可以简化 50%、订阅系统可以删掉、ChromaDB 可以换成 JSON 文件。

- [ ] 是否需要支持多用户 → ❓未验证

### C2. 新技术选了这么多，真的需要吗？

```
本项目的技术依赖:
  Java: Spring Boot, Resilience4j, Mockito, Jackson, Lombok
  Python: FastAPI, ChromaDB, sentence-transformers, CrossEncoder, 
          spaCy, SnowNLP, NLLB-200

如果砍掉一半:
  不需要 ChromaDB        → JSON 文件 + 内存 cosine
  不需要 CrossEncoder    → 只用 Embedding
  不需要 NLLB-200        → LLM 翻译 (AI_PROVIDER 本来就调了)
  不需要 SnowNLP         → LLM 情感分析
  不需要 spaCy           → LLM NER

省下: 模型加载时间、内存、GPU 依赖、维护成本
换来: 每个请求多 200 tokens (LLM 翻译+情感+NER)
```

用户量小的时候，用 LLM 做翻译/情感/NER 比本地模型更便宜（少维护四套本地模型）。但如果用户量大了，本地模型又更便宜。当前阶段应该权衡。

- [ ] ChromaDB 是否可替换 → ⚠️存疑
- [ ] 本地翻译模型是否必要 vs LLM 翻译 → ❓未验证
- [ ] 本地 NER/情感 是否必要 → ❓未验证
